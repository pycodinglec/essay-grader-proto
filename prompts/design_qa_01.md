# 설계 Q&A #1 — 기획 검토 (2026-02-23)

프로젝트 기획의 구현 가능성 및 잠재적 문제점을 검토한 결과와 확정 사항.

## 확정 사항

1. **OCR**: Nano Banana Pro API 사용 확정. 이미지 생성 모델이지만 멀티모달 입력을 통한 OCR 용도로 사용.
2. **최고점 채택**: 의도된 설계. 손글씨 OCR 오류로 인한 학생 불이익을 보상하기 위한 관대한 채점 정책.
3. **중간 수정 없음**: 사용자(채점자)가 결과 파일(report.xlsx)을 직접 수정하는 워크플로우. 소프트웨어 내 중간 수정 기능 제공하지 않음.
4. **ZIP 파일 규칙**: 폴더 포함 시 에러 메시지 표시 후 실행 중단. 파일명에 규칙 없음(힌트로 사용 불가). 학생 식별은 전적으로 OCR 내용에 의존.
5. **3 LLM 비용**: 프로토타입 수준에서 수용. Sonnet 4.6 / Gemini 3 Flash / GPT 5.2 (reasoning level high 미지정)으로 비용 절감.
6. **에러 핸들링**: 채점 도중 에러 발생 시 중간 결과 다운로드 안내 메시지 제공.
7. **임의 중단 버튼**: 제거. 사용자가 업로드 분량을 스스로 조절하는 것으로 대체.
